1 · What is a falling-sand game?

A falling-sand game is a 2D sandbox where every pixel is a tiny particle that obeys simple physics rules. Drop some sand and it piles up; pour water and it spreads and finds its level; add fire and it spreads to anything flammable. There's no goal or score — the fun is in experimenting with how materials interact.

Under the hood it's a cellular automaton: a grid of cells where each cell looks at its neighbors and decides what to do next, many times per second. Simple per-cell rules add up to surprisingly lifelike behavior.

Why it's a great first project: the rules are short, the payoff is visual and instant, and you can keep adding materials forever. It teaches the game loop, the HTML5 canvas, and grid/array thinking — the foundations of most 2D games.

2 · Playing sand-game.com

sand-game.com is a free browser version you can open right now — nothing to install. The basics:

Controls

  • Click / drag on the canvas to place the selected material.
  • Pick a material from the palette (sand, water, wall, fire, etc.).
  • Brush size changes how much you drop at once.
  • Clear / reset wipes the canvas.

Things to try

  • Build walls to make a funnel, then pour sand through it.
  • Drop water and watch it level out.
  • Set plants/wood on fire and watch it spread.
  • Mix materials and see what reacts.

Study what it does, then rebuild a small version yourself — that's the fastest way to actually understand it.

3 · How it works (the idea)

Everything comes down to four pieces:

  • A grid — a 2D array. Each cell holds a number/ID for what's there (0 = empty, 1 = sand, 2 = water…).
  • Rules — for each cell, "if the spot below is empty, fall into it." Sand falls straight or diagonally; water also slides sideways.
  • A game loop — update every cell, then draw the grid, ~60 times per second.
  • Input — the mouse "paints" material IDs into the grid.

The one rule that makes sand "fall"

For each sand cell, check the cell directly below. If it's empty, move down. Otherwise try down-left or down-right. If all are blocked, stay put. That's it — pile-forming behavior emerges on its own.

One gotcha: scan the grid from the bottom up. If you scan top-down, a falling grain can move down and then get processed again in the same frame, teleporting to the floor. Bottom-up updating fixes it.

4 · Getting started: project setup

You need nothing but a text editor and a browser. Here's the suggested way to begin, keeping everything in your gaming/html/ folder.

  1. Make a file in this folder, e.g. my-sand-game.html.
  2. Start from the skeleton below — one file holds HTML, CSS, and JS so there's nothing to wire up.
  3. Open it in a browser by double-clicking the file. To see changes, save and refresh.
  4. Iterate in small steps — get a single grain falling before you add water, fire, or a palette.
Editor tip: VS Code (which you're in) + the "Live Server" extension auto-refreshes the page every time you save — much faster than manual refreshing.

5 · Build it step by step

Step 1 — Create the grid

One flat array is fastest. Store a material ID per cell; 0 means empty.

const W = cv.width, H = cv.height;
const EMPTY = 0, SAND = 1, WALL = 2;
let grid = new Uint8Array(W * H);          // all zeros = all empty
const idx = (x, y) => y * W + x;           // 2D -> 1D helper

Step 2 — Make sand fall (the core rule)

Loop bottom-to-top. Each grain tries down, then down-left, then down-right.

function update(){
  for (let y = H - 2; y >= 0; y--){        // bottom-up, skip last row
    for (let x = 0; x < W; x++){
      if (grid[idx(x,y)] !== SAND) continue;
      const below = idx(x, y+1);
      if (grid[below] === EMPTY){ move(idx(x,y), below); }
      else {
        const dir = Math.random() < 0.5 ? -1 : 1;   // try a random diagonal
        const dl = (x+dir >= 0 && x+dir < W) ? idx(x+dir, y+1) : -1;
        if (dl !== -1 && grid[dl] === EMPTY) move(idx(x,y), dl);
      }
    }
  }
}
function move(from, to){ grid[to] = grid[from]; grid[from] = EMPTY; }

Step 3 — Draw the grid

Write colors straight into pixel data — the fastest way to paint a whole grid.

const colors = { [SAND]:[230,192,104], [WALL]:[120,130,145] };
function draw(){
  const img = ctx.createImageData(W, H);
  for (let i = 0; i < grid.length; i++){
    const c = colors[grid[i]] || [0,0,0];
    img.data[i*4]   = c[0];
    img.data[i*4+1] = c[1];
    img.data[i*4+2] = c[2];
    img.data[i*4+3] = 255;
  }
  ctx.putImageData(img, 0, 0);
}

Step 4 — Paint with the mouse

let painting = false, brush = SAND;
function paintAt(e){
  const r = cv.getBoundingClientRect();
  const x = Math.floor((e.clientX - r.left) / r.width  * W);
  const y = Math.floor((e.clientY - r.top ) / r.height * H);
  if (x>=0 && x<W && y>=0 && y<H) grid[idx(x,y)] = brush;
}
cv.addEventListener('mousedown', e => { painting = true; paintAt(e); });
cv.addEventListener('mousemove', e => { if (painting) paintAt(e); });
window.addEventListener('mouseup', () => painting = false);
That's a complete sand game. Drop those four steps into the skeleton's <script> and you'll have falling, piling sand you can paint. Everything after this is just more materials and nicer UI.

Step 5 — Add water (your second material)

Water is sand that also slides sideways when it can't fall — that one extra rule makes it behave like a liquid.

// after the "fall" checks fail for a WATER cell, try moving straight sideways:
const side = Math.random() < 0.5 ? -1 : 1;
const s = (x+side >= 0 && x+side < W) ? idx(x+side, y) : -1;
if (s !== -1 && grid[s] === EMPTY) move(idx(x,y), s);

7 · Ideas to extend it

More materials

  • Stone/wall — never moves (already in the demo).
  • Fire — spreads to flammable neighbors, then burns out.
  • Wood/plant — static but flammable.
  • Steam — water + fire; rises instead of falls.
  • Acid — dissolves whatever it touches.

Polish & features

  • Adjustable brush size.
  • Density rules so sand sinks through water.
  • Color variation per grain for texture.
  • Pause / step buttons for debugging.
  • Save/load a scene to localStorage.
Performance note: a 160×120 grid is ~19k cells and runs easily. If you scale up and it stutters, that's expected — optimize later (e.g. only update "active" regions). Get it working first.
